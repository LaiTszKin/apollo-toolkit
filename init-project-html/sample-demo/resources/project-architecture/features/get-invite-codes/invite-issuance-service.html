invite-issuance-service

自身職責：在同一交易內判斷使用者額度、組裝待寫入列、處理 UNIQUE 碰撞重試。本頁只列自身函式 I/O 與內部資料流。

函式 I/O

函式簽名副作用用途
IssueInviteCode in: ctx, userID
out: InviteCodeRecord | error 		tx 開／關交易
db 條件成立時寫入新行		 對外唯一入口。
checkQuota in: used, limit, userStatus
out: true | ErrQuotaExceeded | ErrForbidden 		pure 窗口計數 + 帳號狀態合法性判斷。
buildRecord in: userID, code: string, now
out: InviteCodeRecord 		pure 組裝待寫入承載：owner_user_id、allowed_registration=true、created_at=now。
shouldRetryOnCollision in: err, attempt: int, max: int
out: bool 		pure err 為 UNIQUE 違例且 attempt < max 才回 true。

變數與業務用途

本表盤點額度判定、寫入承載組裝、與 UNIQUE 碰撞重試所依賴的識別子。

變數型別作用域業務用途
ctx context.Context call 限制整次發放的時間預算；避免重試迴圈無限延長導致連線資源被吃光。
userID UUID call 本次發放歸屬；同時也是額度視窗的主鍵；由 public-api 從可信來源（Principal）注入。
tx *sql.Tx tx 把「額度檢查 + 帳號狀態檢查 + 寫入」綁成原子單元；避免在檢查與寫入之間出現空隙造成雙倍超發。
used int64 call 當前滑動視窗已發放數；額度判定的「左值」；錯算即可能放行超額或誤拒。
limit int instance 滑動視窗額度上限；產品設定，跨呼叫不變；改值代表產品策略改變，需配套更新文案。
userStatus active | banned | suspended call 帳號是否處於可發放狀態；停權或暫停狀態下立即拒絕，不浪費 generator/DB 資源。
now time.Time call 滑動視窗起點計算與 created_at 的基準；測試可注入。
attempt int loop UNIQUE 碰撞重試計數；搭配 max 限制最壞情況避免無限循環。
max int instance 重試上限；超出即拋 ErrCollisionExhausted；上限與 generator 的熵預算共同決定罕見失敗率。
candidate string loop 一輪重試取得的候選碼；尚未確定唯一；UNIQUE 違例即丟棄並重抽，不會被使用者看到。
InviteCodeRecord {code, owner_user_id, allowed_registration, created_at} loop 寫入 DB 的承載；allowed_registration=true 是「邀請碼註冊」功能消費的關鍵旗標——關閉此值代表暫停發放但保留歷史紀錄。

內部資料流

  1. 取得 userID + 開 tx → 讀{used, userStatus}（透過 DB 子模組；其 SQL 介面見宏觀圖邊 e-B-...）。
  2. checkQuota 不通過 → 拋 ErrQuotaExceeded／ErrForbidden；本函式直接結束（呼叫端負責回滾）。
  3. candidate ← 生成器產出的字串。
  4. buildRecord(userID, candidate, now) → InviteCodeRecord。
  5. 嘗試寫入；若 err 與 shouldRetryOnCollision(err, attempt, max) 為 true，attempt+=1，回到步驟 3。
  6. 寫入成功 → 提交 → 回傳 InviteCodeRecord。

本子模組會拋出的錯誤

錯誤條件
ErrQuotaExceededcheckQuota 失敗（含 retryAfter）
ErrForbidden帳號停用
ErrCollisionExhausted重試次數用盡；極罕見，視熵設計。
ErrTransient連線中斷／ctx 取消
