postgresql — 發放側操作

自身職責：對 invite_codes 表執行計數與插入。寫入的列會在「邀請碼註冊」功能裡被另一個視角讀取（跨功能資料流見宏觀架構圖）。本頁只列自身 SQL 函式的 I/O。

SQL 函式 I/O

函式簽名／SQL副作用用途
CountInvitesSince in: tx, userID, since
out: int64
SQL: SELECT count(*) FROM invite_codes WHERE owner_user_id=$1 AND created_at > $2 		read 滑動視窗計數，給呼叫端做額度判斷。
InsertInviteCode in: tx, InviteCodeRecord
out: InviteID | UniqueViolation
SQL: INSERT INTO invite_codes (code, owner_user_id, allowed_registration, ...) VALUES (...) RETURNING id 		write 新列（待 commit） invite_codes.code UNIQUE；衝突時呼叫端負責換碼重試。
GetUserStatus in: tx, userID
out: UserStatus | NotFound
SQL: SELECT status, banned_at FROM users WHERE id=$1 		read 給呼叫端做帳號狀態檢查。

變數與業務用途

本表分兩段：上半為 SQL 函式參數／回傳，下半為 invite_codes / users 表上的關鍵列欄位（被本子模組寫入或讀取）。

變數型別作用域業務用途
tx *sql.Tx tx 上層管理的隔離單元；本子模組所有 SQL 都掛在這個 tx 上以保證原子寫入。
userID UUID call SQL 參數；標記新邀請碼擁有者，也是計數視窗鍵；錯誤帶入會把額度算到別人頭上。
since timestamp call 滑動視窗起點；決定 count(*) 的時段；起點漂移會讓額度執行不準。
InviteCodeRecord {code, owner_user_id, allowed_registration, ...} call 待插入的邀請碼資料；其欄位直接落在 invite_codes 列上。
UserStatus {status, banned_at} call GetUserStatus 的回傳；給呼叫端判定是否准予發放。
invite_codes.code text UNIQUE persist 列上欄位；UNIQUE 約束在本子模組是「衝突信號源」，呼叫端據此換碼重試。
invite_codes.owner_user_id FK → users.id persist 列上欄位；建立「誰擁有這張邀請」的關聯；額度查詢與後續審計都依此欄。
invite_codes.allowed_registration boolean persist 列上欄位；發放時寫 true；「邀請碼註冊」功能在判定時讀此欄，false 即拒絕該邀請可用。
invite_codes.consumed_at timestamp NULL persist 列上欄位；本子模組寫入時為 NULL；之後由「邀請碼註冊」標記消費，是橋接兩個功能模組的關鍵欄位。
invite_codes.created_at timestamp persist 列上欄位；額度的滑動視窗計數憑此欄位，業務上要與 now 同步以免漂移。

內部資料流（單一交易視角）

  1. 呼叫端開 tx → 本子模組以該 tx 執行 SQL；交易控制不在本子模組內。
  2. CountInvitesSince 與 GetUserStatus 為純讀；不變動列。
  3. InsertInviteCode：寫入後回傳 InviteID 或 UniqueViolation；無 partial row。
  4. 所有寫入皆在 tx 內；本子模組對外只報告每個 SQL 結果。

錯誤觸發 SQL
