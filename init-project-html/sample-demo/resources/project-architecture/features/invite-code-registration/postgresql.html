postgresql — 註冊側操作

自身職責：以 SQL 對 invite_codes／users 表執行讀鎖、寫入與條件更新。本子模組同時是讀者（查邀請碼）與寫者（建 user、核銷邀請）。本頁只列自身 SQL 函式的 I/O。

SQL 函式 I/O

函式簽名／SQL副作用用途
GetInviteForUpdate in: tx, code: text
out: InviteRow | NotFound
SQL: SELECT … FROM invite_codes WHERE code=$1 FOR UPDATE 		lock 對該列加 row lock 直到 tx 結束 取得邀請碼狀態並阻止他人並行核銷。
InsertUser in: tx, {email, passwordHash, inviteID}
out: UserID | UniqueViolation
SQL: INSERT INTO users (...) RETURNING id 		write 新列（待 commit） 建立使用者列；users.email UNIQUE。
MarkInviteConsumed in: tx, inviteID, userID, now
out: RowsAffected
SQL: UPDATE invite_codes SET consumed_at=$3, consumed_by=$2 WHERE id=$1 AND consumed_at IS NULL 		write 條件更新 標記核銷；RowsAffected=0 表示已被搶先核銷。

變數與業務用途

本表分兩段：上半為 SQL 函式參數，下半為 invite_codes / users 表上的關鍵列欄位（被本子模組讀寫）。

變數型別作用域業務用途
tx *sql.Tx tx 上層管理的隔離單元；本子模組所有 SQL 都掛在這個 tx 上，未 commit 即視為「無事發生」，保證註冊原子性。
code text call GetInviteForUpdate 的查鎖鍵；對應到列上欄位 invite_codes.code 的 UNIQUE 索引，避免並行核銷相同邀請。
inviteID UUID tx 從 SELECT 取得後跨多個寫入；把「核銷哪一張邀請」與「新 user 屬於哪張邀請」綁定，作為日後審計鏈。
userID UUID tx InsertUser RETURNING id 的結果；同一 tx 內回填到 consumed_by，完成「誰用了邀請」的紀錄。
now timestamp call 核銷時的官方時間；由應用層提供以避免應用與 DB 時鐘漂移影響審計順序。
RowsAffected int call MarkInviteConsumed 條件更新結果；0 表示已被搶先核銷，本次必須回滾，否則會發生雙倍消費。
consumed_at timestamp NULL persist invite_codes 列上欄位；非空代表「這張邀請已被消費」，與 WHERE consumed_at IS NULL 一起防雙重核銷。
consumed_by FK → users.id persist invite_codes 列上欄位；紀錄是哪個 user 用掉這張邀請，是「邀請碼註冊」這條業務路徑留下的最終痕跡。
users.email text UNIQUE persist 使用者唯一鍵；UNIQUE 違例代表「相同信箱已註冊」，業務上以 409 回應，呼叫端不應靜默忽略。

內部資料流（單一交易視角）

  1. 呼叫端開 tx → 本子模組以該 tx 執行 SQL；交易控制不在本子模組內。
  2. GetInviteForUpdate 取得 InviteRow 並把 列鎖 綁在 tx 上。
  3. InsertUser 取得 UserID，作為下一步 consumed_by 的 FK 值。
  4. MarkInviteConsumed 以 WHERE consumed_at IS NULL 防雙重核銷。
  5. 所有寫入皆在 tx 內；commit／rollback 由呼叫端決定，本子模組對外只報告每個 SQL 的結果。

錯誤觸發 SQL含義
