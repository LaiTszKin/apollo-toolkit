web-register-ui

自身職責：渲染註冊表單、客戶端驗證、發出 HTTP 請求、把結果映射成 UI 狀態。本頁只談自身函式 I/O 與資料流。

函式 I/O

函式簽名副作用用途
useRegisterForm() in: —
out: {values, errors, setField, submit} 		state React state hook 表單狀態容器。
validateClientSide in: FormValues
out: FieldErrors | null 		pure 必填、email 格式、密碼長度等規則。
submit in: FormValues
out: Result<NewUserID, UiError> 		network fetch POST 送出後驅動 loading / success / error UI。
mapApiError in: ApiError{code, message}
out: UiMessage{lang_tag, text} 		pure 機讀錯誤碼轉成本地化訊息。

變數與業務用途

本表盤點本子模組在記憶體中持有、決定 UI 行為的關鍵識別子。

變數型別作用域業務用途
FormValues {email, password, inviteCode} instance 使用者目前打算送出的註冊資料；驅動所有後續驗證、按鈕可用性與請求 body。
FieldErrors Record<field,msg> | null call 客戶端驗證結果；非空時阻擋網路請求，避免明顯無效嘗試打到後端造成不必要負擔與錯誤訊息。
UIState idle | loading | error | success instance UI 機器狀態；loading 期間禁用送出按鈕避免重複提交；其他狀態決定要顯示哪個訊息／導向。
Result {ok: NewUserID} | {err: UiError} call 一次 submit 的最終回應；驅動「導向後續頁」或「停留修改再試」兩條業務分支。
UiMessage {lang_tag, text} call 後端機讀錯誤碼的本地化呈現；讓使用者用自身語言理解失敗原因（例如「邀請碼已被使用」）。

內部資料流

  1. 使用者輸入 → setField 改寫 FormValues。
  2. 送出時 validateClientSide(FormValues) 若回 FieldErrors，UI 設為 error，不觸發網路請求。
  3. 通過驗證後 submit(FormValues)：UI 轉 loading，等待 Result。
  4. Result.ok → UI 轉 success，呼叫導向；Result.err → 經 mapApiError 變 UiMessage 顯示。
  5. 失敗後 FormValues 保留，使用者可修正並再次 submit。
