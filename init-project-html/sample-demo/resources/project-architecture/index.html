Acme App — 功能模組 × 子模組（同一張圖）

更新於 2026-05-11（UTC） · 此頁同時呈現「功能模組之間」與「子模組之間」的多對多交互；子模組頁本身只描述自身函式 I/O 與內部資料流。

為何把兩層放同一張圖

功能模組（對使用者可見的能力）的「宏觀交互」實際上是由其底下的子模組互相呼叫與資料交換達成的。為了讓讀者能在同一張圖同時看到「功能 A → 功能 B」與「達成這條路徑的具體子模組是誰呼叫誰、誰讀誰寫」，本宏觀圖以兩個功能模組為群組（cluster），群組內畫出子模組節點與彼此的呼叫、回傳與寫讀資料邊。

一個子模組可能同時是生產者與消費者。本示範刻意保留：postgresql 的 invite_codes 表在「獲取邀請碼」功能裡被寫入，產生的列在「邀請碼註冊」功能裡被讀取／核銷；registration-service 對 postgresql 同時是讀者（SELECT FOR UPDATE）與寫者（INSERT user / UPDATE invite consumed）；invite-issuance-service 把 invite-code-generator 當成多次呼叫的內嵌函式並回收其結果。

圖例

  • 虛線大方框：一個功能模組（feature module）= 對使用者可見的能力。標題即連到該功能總覽。
  • 實心小方塊：子模組。每一塊點擊進入該子模組頁（內容只談自身函式 I/O 與資料流）。
  • 實線箭頭：函式呼叫／HTTP 請求（caller → callee）。
  • 虛線細箭頭：回傳值／HTTP 回應（callee → caller）。
  • 暖色粗虛線：跨功能資料行級別的生產者→消費者關係（與函式呼叫不同：透過 DB 行傳遞）。
  • 同一對節點可有多條邊：例如 registration-service 對 postgresql 有 SELECT、INSERT、UPDATE 三條獨立邊。
